# PaddleOCR vs ONNX Runtime OCR ベンチマーク比較レポート

## 概要

PaddleOCR (`ocr.py`) と ONNX Runtime (`ocr_onnx.py` / `ocr_onnx_single.py`) の OCR 推論パイプラインを、
同一画像セット・同一解像度で比較した。
推論速度・VRAM 使用量・テキスト検出精度の3軸で評価し、ONNX 版の VRAM スパイクの原因と緩和策を検証した。

## 環境

| 項目 | 値 |
|------|-----|
| GPU | NVIDIA GeForce RTX 3080 Ti (12,288 MiB) |
| PaddleOCR | v3.4.0 (PaddlePaddle GPU 3.2.x) |
| ONNX Runtime | onnxruntime-gpu 1.24.1 (CUDA EP) |
| モデル | PP-OCRv5 server (det + rec) + PP-LCNet cls |
| テスト画像 | `debug_kie/` から20枚、長辺2000pxに統一 (約2000x283) |

## 比較条件

5つの構成を比較した。

| 構成 | 検出方式 | セッション管理 | ファイル |
|------|---------|---------------|---------|
| PaddleOCR | 1枚推論 | エンジン再利用 | `ocr.py` |
| ONNX-A: SAHI+毎回生成 | SAHIパッチ推論 | 毎回再生成 | `ocr_onnx.py` |
| ONNX-B: 1枚+キャッシュ | 1枚推論 | セッション保持 | `ocr_onnx_single.py` (v1) |
| ONNX-C: 1枚+2フェーズ | 1枚推論 | det→破棄→rec+cls→破棄 | `ocr_onnx_single.py` (v2) |
| ONNX-D: バッチ2フェーズ | 1枚推論 | 全画像det→破棄→全画像rec→破棄 | `ocr_onnx_single.py` (v3) |

ONNX-C, ONNX-D では以下のプロバイダオプションも適用:

```python
("CUDAExecutionProvider", {
    "arena_extend_strategy": "kSameAsRequested",
    "cudnn_conv_use_max_workspace": "0",
    "cudnn_conv_algo_search": "HEURISTIC",
})
```

## 結果サマリ

### 推論速度

| 指標 | PaddleOCR | ONNX-A | ONNX-B | ONNX-C | ONNX-D |
|------|-----------|--------|--------|--------|--------|

| PaddleOCR 比 | 1.00x | 4.9x 遅い | 1.28x 遅い | 3.37x 遅い | **1.42x 遅い** |
| スループット | **4.58枚/秒** | 0.94枚/秒 | 3.52枚/秒 | 1.37枚/秒 | 3.22枚/秒 |

### モデルロード

|------|-----------|------------|
| ロード時間 | 2.58秒 | **0.36秒 (7.2倍速い)** |
| モデル VRAM | 476 MiB | **300 MiB (37%軽い)** |

### ピーク VRAM 使用量

| 指標 | PaddleOCR | ONNX-A | ONNX-B | ONNX-C | ONNX-D |

## VRAM スパイクの原因分析

### ステージ別 VRAM 増分（ONNX 単体、2000x283 画像1枚）

| ステージ | VRAM (絶対) | ベースラインからの増加 |
|----------|-----------|---------------------|
| ベースライン | 445 MiB | — |
| Det セッションロード | 810 MiB | +364 MiB |
| Det 推論後 | 1,462 MiB | **+1,017 MiB** |
| `del det_session` + GC 後 | 712 MiB | +266 MiB |
| Rec + Cls セッションロード | 824 MiB | +378 MiB |

`del det_session` により **VRAM は確実に解放される** (1,462 → 712 MiB)。

### 根本原因 1: 検出モデルの中間活性化マップ

検出モデル (PP-OCRv5_server_det) の ResNet バックボーン初期段階で、
64チャネルの特徴マップ結合（`Concat.16` ノード）が発生する。
2000x283 の入力（テンソル 6.5MB）でも中間特徴量マップの展開で **+1.0GB** を消費する。

> 参考: 3509x4963 画像では `Concat.16` が 2.73GB を要求し OOM で失敗した。

### 根本原因 2: ONNX Runtime のメモリアリーナが解放されない

| 項目 | ONNX Runtime | PaddlePaddle |
|------|-------------|-------------|
| アリーナ拡張戦略 | `kNextPowerOfTwo`（2のべき乗で過剰確保） | 統一メモリプール |
| セッション間メモリ共有 | **不可**（det/rec/cls で3つのアリーナ） | 全モデルで共有 |
| cuDNN ワークスペース | 最大確保 | 必要最小限 |
| メモリ解放タイミング | **セッション破棄まで保持** | 推論ステージ間で再利用 |

PaddlePaddle は det が使い終わったメモリを rec が再利用できる。
ONNX Runtime は det のアリーナと rec のアリーナが独立しているため、同時保持すると合算になる。

### 根本原因 3: rec アリーナの単調増加問題（バッチ処理時）

バッチ2フェーズ (ONNX-D) の検証で判明した問題。rec セッションを保持したまま
**異なるサイズのバッチを逐次投入すると、アリーナが単調に膨張する**。

```
rec 1画像目 (37 crops):   2,414 MiB (+1,968)
rec 全20画像 (1,323 crops): 12,124 MiB (+11,678)  ← 膨張し続ける
del rec_session 後:          718 MiB (+272)        ← 解放される
```

ONNX Runtime のアリーナは「過去の最大確保を保持しつつ、新しいサイズが必要なら追加確保」するため、
画像ごとにクロップ数（= バッチサイズ）が変わると最大サイズ分のメモリが累積する。
これは `kSameAsRequested` でも防げない。

### SAHI がむしろ VRAM 節約に寄与していた

| | SAHI あり (960px タイル) | SAHI なし (2000px 全体) |
|--|------------------------|----------------------|
| Det 入力サイズ | 小 (タイル単位) | 大 (画像全体) |
| 中間特徴量 | 小 | **大** |
| ピーク VRAM | 1,964 MiB | 7,276 MiB |

SAHI はタイル単位で検出モデルに投入するため、中間特徴量のサイズが抑制される。

## 緩和策と実測効果

### 1. プロバイダオプション変更

```python
"arena_extend_strategy": "kSameAsRequested"
"cudnn_conv_use_max_workspace": "0"
"cudnn_conv_algo_search": "HEURISTIC"
```

**効果: 約 200 MiB 削減**。根本解決にはならない。

### 2. 2フェーズ方式（det→破棄→rec+cls、画像単位）

det と rec+cls のセッションを同時に保持せず、画像ごとに生成・破棄する。

| 指標 | ONNX-B (セッション保持) | ONNX-C (2フェーズ) |
|------|------------------------|-------------------|
| ピーク VRAM | 7,276 MiB | **1,588 MiB** |
| 平均推論時間 | 0.284秒 | 0.728秒 |

**VRAM は PaddleOCR 同等まで削減**。ただしセッション生成コスト（約0.45秒/画像）で速度が 2.6 倍悪化。

### 3. バッチ2フェーズ方式（全画像一括）

セッション生成を2回に抑えて速度を改善する狙い。

| 指標 | ONNX-C (画像単位) | ONNX-D (バッチ) |
|------|-------------------|----------------|
| 平均推論時間 | 0.728秒 | **0.311秒** |
| ピーク VRAM | 1,588 MiB | **膨張（rec アリーナ累積）** |

速度は改善するが、rec のアリーナ累積問題により **VRAM が単調増加する**。
20画像で 12GB に達し、実用上は使えない。

### 4. SAHI の併用

大きな画像に対しては SAHI を適用し、検出モデルへの入力サイズを制限する。
VRAM は 2GB 未満に抑えられるが、推論速度は 5 倍程度悪化する。

## 各構成の比較表

| 構成 | 平均推論 | ピーク VRAM | VRAM 安定性 | 備考 |
|------|---------|-----------|------------|------|
| PaddleOCR | **0.218秒** | **1,576 MiB** | 安定 | ベースライン |
| ONNX-A: SAHI | 1.019秒 | 1,964 MiB | 安定 | VRAM 良好だが遅い |
| ONNX-B: キャッシュ | 0.284秒 | 7,276 MiB | 安定（高止まり） | 速いが VRAM 爆食い |
| ONNX-C: 画像単位2フェーズ | 0.728秒 | **1,588 MiB** | **安定** | VRAM 最良だが遅い |
| ONNX-D: バッチ2フェーズ | 0.311秒 | 膨張 | **不安定** | rec アリーナ累積 |

## 結論

**ONNX Runtime で速度と VRAM を両立する構成は現時点では存在しない。**

| 観点 | PaddleOCR | ONNX Runtime (最良構成) |
|------|-----------|----------------------|
| 推論速度 | **0.218秒** | 0.284秒 (ONNX-B) / 0.311秒 (ONNX-D) |
| VRAM 効率 | **1,576 MiB** | 1,588 MiB (ONNX-C) |
| 速度 + VRAM 両立 | **可能** | **不可能**（速度優先なら VRAM 爆発、VRAM 優先なら速度低下） |
| モデルロード | 2.58秒 | **0.36秒** |
| デプロイ容易性 | PaddlePaddle 依存 | **ONNX 標準フォーマット** |

### 推奨構成の選び方

- **速度最優先・VRAM 余裕あり** → ONNX-B（セッション保持、ピーク 7.3GB）
- **VRAM 制約厳しい・速度許容** → ONNX-C（画像単位2フェーズ、ピーク 1.6GB）
- **バランス重視** → PaddleOCR（速度・VRAM ともに最良）
- **PaddlePaddle 依存を避けたい** → ONNX-C + 入力サイズ制限の併用

### 今後の改善余地

- ONNX Runtime の `MemoryArena` を複数セッションで共有する仕組み（現時点では未サポート）
- rec モデルへの入力テンソルサイズを統一パディングし、アリーナの再割り当てを防止する
- ONNX Runtime の `OrtCUDAProviderOptionsV2` に `gpu_mem_limit` を設定して上限を設ける
